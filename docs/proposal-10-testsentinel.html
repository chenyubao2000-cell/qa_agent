Proposal-10: TestSentinel

多平台质量守卫 — 7×24 全时监控 · 主动触发行动

📡 质量守卫模式 ✅ 被动监控零负担 🤖 Agent 55% ⚠ 监控补充层

🎯 核心理念

不聚焦"生成测试"，聚焦"监控质量"

TestSentinel 将 Agent 定位为全时在线的质量守卫，而非按需调用的测试生成器。 它持续监控所有平台的质量信号——错误日志、LLM 评分趋势、部署状态、Schema 变更、PR 差异—— 一旦检测到异常，立即分析根因并主动触发行动：生成回归测试、通知团队、建议回滚。

❌ 传统模式

开发者发现问题 → 手动触发 Agent → 生成测试 → 被动响应

✅ TestSentinel 模式

Agent 持续监控 → 自动发现异常 → 分析根因 → 主动触发行动 → 通知团队

👀 Agent 守卫模式 — 7×24 运行

🐛
Sentry Watcher
每 5 分钟轮询
持续轮询 Sentry 新 issue，发现异常后自动分析 stack trace，定位源码中的问题代码，生成对应的回归测试用例。
poll sentry/issues?status=unresolved
→ parse stack_trace
→ locate source code
→ generate regression test
→ notify team
🧠
Langfuse Watcher
每小时 eval
每小时拉取 Langfuse trace 数据，监控 LLM 输出质量趋势，当评分低于阈值时自动触发详细 eval 并通知团队。
fetch langfuse/traces?last=1h
→ compute quality score
→ compare with threshold
→ trigger deep eval if anomaly
→ alert team
🚂
Railway Watcher
部署事件驱动
监听 Railway 部署事件 webhook，部署完成后立即运行 smoke 测试，分析结果并建议回滚或继续。
receive deploy webhook
→ wait for healthy status
→ run smoke tests
→ analyze results
→ recommend rollback / proceed
🗄
DB Watcher
migration 监听
监听 drizzle-kit diff 输出，检测 schema 变更后自动生成数据兼容性测试，确保迁移不破坏既有功能。
detect drizzle-kit diff
→ parse schema changes
→ generate compat tests
→ execute & report
💻
PR Watcher
复用 qa_agent git-watcher
复用已有的 git-watcher 脚本，监听 PR 变更事件，分析代码差异，给出测试建议与影响评估。
git-watcher detects PR event
→ analyze diff
→ suggest test coverage
→ impact assessment
→ comment on PR

⚖ 优势与劣势

✅ 优势

  • 被动监控不增加开发者负担，零打扰运行
  • 覆盖所有监控平台，形成统一质量视图
  • Agent 主动发现问题，而非被动等待触发
  • 可作为其他方案的监控补充层，兼容性强
  • 7×24 全时运行，不遗漏任何质量信号

⚠ 劣势

  • 需长时间运行 Agent，产生持续的 API 调用成本
  • 误报可能产生噪音，需要精心调校阈值
  • 依赖各平台 API 稳定性与 webhook 可靠性
  • 多 Watcher 并行运行时的资源协调

🎯 适用场景

TestSentinel 最适合作为其他方案的监控补充层。它不替代主动的测试生成流程， 而是在所有测试方案之上增加一道全时在线的质量防护网。 无论团队采用哪种测试生成策略（TestPilot、PRD 驱动、Issue 驱动等）， TestSentinel 都能在后台持续监控各平台的质量信号，确保任何异常都不会被遗漏。

特别适合：已有测试体系但缺乏统一监控的团队 · 多平台部署的复杂系统 · 对 LLM 输出质量有持续监控需求的 AI 产品
