2 架构流水线

从 GitHub PR Event 出发，经过增量分析、智能生成、选择性执行，到最终覆盖报告。

TRIGGER

GitHub PR Event

开发者提交 PR，触发 CI 流水线。所有后续步骤围绕此 PR 展开。
AUTO

增量分析引擎

Diff Analyzer
解析 PR 中每个文件的变更行与变更类型
Dependency Graph
构建模块间 import 依赖图谱
Test Impact Analyzer
基于依赖图确定受影响的测试集
Coverage Gap Finder
对比 coverage diff 识别未覆盖的新增/修改代码
AGENT HUMAN

Incremental Test Generator

Agent 分析 diff + 覆盖缺口，按现有测试风格生成测试建议，以 PR Comment 形式推送。
支持三种生成策略：AI 生成 / Template 匹配 / PR Comment 建议
▶ 人工审核：查看 PR comment 中的建议 → 一键 approve 或编辑后 approve → 自动 commit 到 PR
AUTO

Selective Test Runner

仅运行受影响的测试包（Affected packages only），基于 Turborepo affected + import graph 确定范围。
大幅减少 CI 执行时间，同时保证变更区域的测试覆盖。
AUTO

覆盖报告 + 质量门禁

PR comment 自动更新覆盖率变化。当 coverage delta < 阈值 时自动 block PR。
人工设定质量门禁规则（覆盖率阈值、必须通过的测试集等）
AGENT

失败分析 + 修复建议

Agent 分析失败原因，区分 Flaky / 真失败 / 环境问题。
提供修复方案，人决定是否采纳。

4 与 qa_agent 的协同

TestFlow 在 CI 层负责单元测试的增量覆盖，qa_agent 在 Agent 层负责 E2E 测试，两者通过 GitHub PR 为纽带协同。

CI 层

TestFlow

单元测试 / 增量覆盖
变更影响分析 / 质量门禁

↔ GitHub PR
Agent 层

qa_agent

E2E 测试 / 页面探查
CDP 驱动 / Linear 汇报

协同方式： TestFlow 检测到 PR 中的变更后，在 CI 流水线中完成单元测试增量覆盖。qa_agent 通过 /qa-from-branch 从同一 PR 驱动 E2E 测试。两者结果均以 PR comment 形式呈现，人在同一个 PR 页面完成审核。
5 方案评估
Agent 占比
35%

✓ 优势

  • 最低引入成本，可快速落地到现有 CI 流程
  • 人始终在 loop 中，保持对测试质量的掌控
  • 与 qa_agent 天然互补，CI 层 + Agent 层分工明确
  • 增量分析大幅减少无关测试的执行时间

✗ 劣势

  • 人工审核是瓶颈，高频 PR 场景下可能积压
  • 增量分析可能遗漏间接依赖导致的回归
  • Agent 生成的测试建议质量依赖上下文完整度

🎯 适用场景

快速落地、人力有限但希望保持控制的团队。适合中小规模项目，PR 频率适中，团队希望逐步引入 AI 辅助测试而非一步到位的全自动化。
