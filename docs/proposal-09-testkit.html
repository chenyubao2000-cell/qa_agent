◆ 核心理念

TestKit 不预设"正确答案"，而是提供一套 Plugin 协议和组合工具链。 团队从 Plugin Registry 中选取所需积木，像拼装乐高一样构建自己的测试基础设施。 Agent 在此架构中不是控制者，而是可选的、可替换的 Plugin 之一。

Plugin-first
一切能力以 Plugin 形式存在：生成、Mock、执行、报告、监控。无 Plugin 则无功能，按需加载。
Agent = Plugin
Agent 不是架构的"大脑"，而是可插拔的 Plugin。团队可以选择纯自动化路径（0% Agent），也可以让 Agent 参与每一步。
Config-driven
一个 testkit.config.ts 文件定义所有 Plugin 组合与参数。版本可控、可审计、可复现。

{} Plugin 协议

与 Mira ToolRegistry 一致的接口协议，确保生态互通

// Plugin 协议 — 与 Mira agent-core ToolRegistry 一脉相承

interface TestKitPlugin {
  name: string; // e.g. "@testkit/gen-ai"
  type: 'generate' | 'mock' | 'run' | 'report' | 'monitor';

  // Agent 能力声明（可选）
  agentModel?: string; // e.g. "claude-sonnet-4-20250514"
  agentSkill?: string; // e.g. "code-review"

  // CLI 与 CI 集成
  cliCommand?: string; // 注册到 testkit CLI 的子命令
  ciIntegration?: CIConfig; // GitHub Actions / GitLab CI 配置
}
// testkit.config.ts 示例

export default defineConfig({
  plugins: [
    genAi({ model: 'claude-sonnet-4-20250514' }),
    mockHttp(), // MSW — 零 Agent
    mockLlm({ record: true }), // Agent 录制回放
    runVitest(),
    reportGithub({ pr: true }),
    monitorSentry({ poll: '5m' }),
  ],
});

★ 方案评估

优势

  • 最灵活 — 团队可按实际需求自由组合 Plugin
  • 与 Mira agent-core ToolRegistry 一脉相承，复用已有协议
  • 社区可扩展 — 任何人可发布第三方 Plugin
  • Agent 占比可调 — 从 0% 到 100% 按需设定

劣势

  • 初期 Plugin 生态不足，需要先期投入建设
  • 组合复杂度 — Plugin 间兼容性需要额外验证
  • 新人上手需理解 Plugin 协议与配置体系

适用场景

  • 工程能力强、有定制需求的团队
  • 需要渐进式引入 Agent 能力的组织
  • 多技术栈并存、需统一测试工具链的平台团队
  • 已有 Mira 生态、希望复用 ToolRegistry 的团队
