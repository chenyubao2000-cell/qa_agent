▸ 流水线概览

从 YAML 声明到测试报告的完整闭环

📝

1. 编写测试意图

人用 YAML DSL 描述测试场景、期望结果和 Mock 需求。Agent 提供 Spec 模板建议并补全缺失字段。

⚙

2. Spec 编译

Agent 将 YAML intent 翻译为具体测试代码（Playwright / Jest / k6）。DSL Validator 自动校验语法和语义。

🔌

3. Mock 声明与生成

人在 YAML 中声明 Mock 需求，Agent 自动生成 Mock 配置，自动化负责 Mock Server 的启停。

▶

4. 测试执行

编译后的测试代码由 CI 自动执行，无需人工或 Agent 介入。全自动化完成。

📊

5. 结果映射

Agent 将原始测试结果映射回 YAML Spec 层级，自动化生成可读报告。

↺

6. Spec 维护

人根据需求变化更新 YAML。Agent 检测代码变更并建议 Spec 更新，由 git diff 触发。

▦ 职责矩阵

人 / Agent / 自动化 三方在每个环节的具体分工

一个完整的 TestSpec YAML 声明，覆盖 Unit / Integration / E2E / Performance 四类测试

YAML testspecs/mira-chat-core.testspec.yaml 
# ============================================
# TestSpec - Mira Chat Core
# ============================================

suite: Mira Chat Core
target: apps/mira-work

# --- Mock 声明 ---
mocks:
  llm:
    strategy: recorded_replay      # 录制回放，保证确定性
    fixtures: fixtures/llm/*.json
  database:
    strategy: schema+seed           # 基于 schema 自动建库 + seed 数据
    schema:   prisma/schema.prisma
    seed:     fixtures/db-seed.sql

# --- 测试声明 ---
tests:

  # Unit 测试 - Tier 分级
  unit:
    - name: 消息解析器正确处理 Markdown
      tier: 1                          # Tier 1: 核心路径
      target: src/utils/message-parser.ts
      intent: 输入各种 Markdown 格式，验证解析输出结构正确
      cases:
        - input: "**bold** and *italic*"
          expect: contains [bold_node, italic_node]
        - input: "```code block```"
          expect: contains [code_block_node]

    - name: Token 计数器准确计算
      tier: 2                          # Tier 2: 重要但非关键
      target: src/utils/token-counter.ts
      intent: 验证不同编码下的 token 计数准确性

  # Integration 测试
  integration:
    - name: 发送消息 → 收到 LLM 回复
      intent: 模拟用户发送消息，验证经过 API 层后能收到 LLM 回复
      requires_mock: [llm, database]
      setup: 创建测试用户和对话
      steps:
        - POST /api/chat/send { message: "Hello" }
        - 等待 SSE stream 完成
        - 验证回复内容匹配 recorded fixture

  # E2E 测试 - 委托 qa_agent
  e2e:
    delegate: qa_agent                 # 由 QA Agent 平台接管
    scenarios:
      - name: 新用户首次对话完整流程
        intent: 注册 → 创建对话 → 发送消息 → 收到回复 → 查看历史
      - name: 对话上下文保持
        intent: 多轮对话中 LLM 能引用之前的内容

  # Performance 测试
  performance:
    tool: k6
    scenarios:
      - name: 并发消息发送
        vus: 50
        duration: 30s
        thresholds:
          p95_response_time: < 500ms
          error_rate: < 1%
⚖ 方案评估

优势、劣势与适用场景分析

✓ 优势

意图聚焦 — 人只需关注"测什么"，不需要了解测试框架细节

✓ 优势

可读性高 — YAML 格式直观易懂，PM / QA / 开发都能参与审阅和编写

✓ 优势

可追溯 — YAML Spec 纳入 Git 版本控制，每次变更有迹可循

✗ 劣势

DSL 设计难度 — DSL 的表达能力是核心难点，设计不当会限制整个平台

✗ 劣势

复杂场景受限 — 高度定制化的测试场景可能无法用声明式 YAML 完整表达

★ 适用场景

注重文档化、需求可追溯性的团队 — 尤其适合有合规要求或需要测试审计的项目
