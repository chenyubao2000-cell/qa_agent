/qa-verify-fix <issues>

CMD 命令层 USER 用户交互 LINEAR CDP 页面探查 SONNET 生成/执行
P0 初始化
CMD 读取配置 + 初始化工作区
同 qa-from-issue：.env → projectContext → phase-0-init.md
P1 读取 Issue + 提取验证上下文
LINEAR 获取 Issue 详情
输入
key / URL / UUID / 关键词
输出
issueContext: { pageUrl,
expectedBehavior (断言目标!),
actualBehavior (仅日志),
reproSteps, feature }
若 expectedBehavior 不明确 → 先询问用户
CMD 是否有已有 verify-fix spec？
Mode R: 已有 spec
重写断言为 expectedBehavior
生成新文件 {slug}-verify-fix.test.ts
跳过 CDP 探查
Mode N: 无已有 spec
需新建
进入 P1.5 预检
P1.5 Bug 复现预检（CDP 快速验证）
CDP 复现检查
导航到 pageUrl → 按 reproSteps 逐步操作 → 截图 + DOM 快照
→ 对比观察到的状态与 actualBehavior
CDP 预检结果
复现（Bug 仍在）
报告用户
"Bug 尚未修复"
修复后重新运行
/qa-verify-fix {key}
流水线终止
未复现（Bug 已消失）
USER 继续生成验证用例？
是
继续 P2→P5
否
终止
无法判断
继续完整
验证流水线
P2 CDP 轻量探查（仅 Mode N）
CDP 定向探查（仅定位器发现）
输入
pageUrl, reproSteps, targetArea
输出
baselineFile, 定位器列表
不构建完整状态流图，仅发现元素定位器
P3 生成 verify-fix Spec
SONNET e2e-orchestrator verifyFix: true
输入
source:"issue", issueContext, baselineFile
关键差异 vs qa-from-issue
断言目标 = expectedBehavior
严格断言 === 而非 >=
通过 = 已修复，失败 = 未修复
生成规则
3-8 个用例（聚焦非穷举）
前缀 [VERIFY-FIX]
头部 // mode: verify-fix
输出
{slug}-verify-fix.test.ts
P4 执行测试
SONNET test-executor
输入
mode:"selective", verify-fix spec
输出
verify-fix-results.json
P5 判定 + Linear 报告
CMD 测试结果判定
FIXED（全通过）
LINEAR 评论 + 更新状态
评论 "Bug 修复已验证 ✅"
update_issue → Done
PARTIALLY_FIXED（部分通过）
LINEAR 仅评论
评论 "Bug 修复未通过验证 ❌"
列出失败用例
不改 issue 状态
NOT_FIXED（全失败）
LINEAR 仅评论
评论 "Bug 修复未通过验证 ❌"
Bug 行为仍可复现
不改 issue 状态
CMD 生成 verify-fix-summary.md
Issue、时间、结论、验证结果表、Issue 详情
流水线结束
