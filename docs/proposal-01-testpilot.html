◆ 核心理念

将 qa_agent 现有的 E2E Agent 模式扩展为覆盖全测试类型的 Agent 集群。 每种测试类型配备一个专用 Agent，由 Orchestrator Agent 统一调度， 实现从单元测试到性能测试的全链路自动化覆盖。 充分复用已有的 Command / Agent / Skill 架构，在现有 4 个 Agent 基础上扩展 5 个新 Agent， 形成完整的测试 Agent 集群。

⚓ 技术栈

复用 qa_agent 的 Command / Agent / Skill 架构，集成主流测试工具链

核心架构

  • qa_agent Command 层
  • Agent 调度框架
  • Skill 能力模块

测试框架

  • Vitest -- 单元测试
  • Playwright -- E2E 测试
  • k6 -- 性能测试

  • Hono mock-server（现有）

外部集成

⚗ 黑盒 Mock 策略

Agent 自动分析代码中的外部依赖，生成 mock 配置，实现测试环境的完全隔离

依赖分析
Agent 自动扫描代码，识别外部 API 调用、数据库连接、第三方服务等依赖

WireMock 提供 HTTP 级别的请求模拟与响应录制
LLM 调用
录制回放模式 -- 首次调用录制真实响应，后续测试直接回放
★ 方案评估

▲ 优势

  • 最高程度自动化，覆盖从需求到监控的全测试链路
  • 复用 Mira 已有 Agent 架构，降低开发成本
  • 能理解自然语言需求，从 PRD / Issue 直接生成测试

▼ 劣势

  • LLM 调用成本高，大规模使用需考虑预算
  • 生成质量不稳定，关键用例需人工审核
  • 全栈覆盖但每个领域可能不够深入

▶ 适用场景

  • 中小型团队，QA 人力有限
  • 需求文档完善，PRD 质量较高
  • 希望快速获得测试覆盖，接受迭代优化
