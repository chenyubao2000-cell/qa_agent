Proposal 08

性能工程一体化 TestBench

性能测试不是事后补充，而是开发流程的一等公民。
Agent 负责从需求提取性能指标并自动生成测试。

核心理念
"Performance is a feature, not an afterthought."

传统性能测试往往在开发末期才被引入，导致性能问题发现晚、修复成本高。TestBench 将性能测试左移（shift-left），让 Agent 在需求阶段就提取 SLA 指标，自动生成 k6/Locust/Gatling 脚本，并在每次 PR 中进行基准对比。对于 AI Streaming 应用而言，延迟是核心体验指标，这种前置策略尤为关键。

Mira 特有性能场景

AI Streaming 应用独有的性能测试维度

⏱

TTFT 首 Token 延迟

从请求发出到接收第一个 token 的时间，直接影响用户感知的响应速度。目标 < 200ms。

⚡

Stream Throughput

Streaming 吞吐量 (tokens/s)，衡量 AI 响应的流畅度。低于 20 tok/s 用户可感知卡顿。

⇄

WebSocket 重连延迟

网络抖动后 WebSocket 自动重连的耗时。影响实时协作场景的连续性体验。

☰

Redis / BullMQ 入队延迟

任务从提交到入队的延迟。高并发下队列背压可能导致任务丢失或超时。

∞

SSE Keep-alive 稳定性

长程任务（>5min）的 SSE 连接保活。中间件超时、CDN 断连是常见故障点。

方案评估

优势、劣势与适用定位

▲ 优势

  • 性能 shift-left，问题在 PR 阶段即暴露
  • AI streaming 延迟作为核心体验指标被系统性追踪
  • Git baseline + PR comparison 实现持续性能守护
  • Flame graph 自动分析降低 profiling 门槛

▼ 劣势

  • 功能测试覆盖弱，不能替代 E2E 测试
  • 基础设施成本高（负载生成集群、监控存储）
  • 负载模型设计仍依赖人类经验
  • 初始基线建立需要稳定环境

▶ 适用场景

  • 作为其他方案的性能补充模块
  • 对延迟敏感的 AI Streaming 应用
  • 需要持续性能回归守护的成熟项目
  • 已有功能测试基础，需补齐性能维度
