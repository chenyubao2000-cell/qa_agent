🛡 核心理念

最小化 Agent 依赖，用契约和变异测试实现确定性质量保障。Agent 仅辅助契约提取和结果解读。

契约即防线

从 OpenAPI / TypeScript Type / Drizzle Schema / gRPC Proto / GraphQL 中提取契约，形成不可逾越的质量防线。契约是自动化测试的唯一真理来源。

变异即验证

通过 Mutation Testing 验证测试套件的有效性。如果测试无法杀死变异体，则测试本身存在缺陷。这是测试质量的终极度量。

确定性优先

所有核心环节（契约测试生成、变异测试执行、质量门禁）均为确定性自动化流程。Agent 不参与决策，仅在边缘提供辅助。

⚙ 架构流水线

三阶段流水线：契约提取 → 测试生成 → 变异验证。每一阶段都有明确的输入、输出和确定性保证。

🤖 Agent 极简介入

Agent 只在两个精确切入点介入，其余全部为确定性自动化流程。

切入点 1: 契约提取

Agent 从 TypeScript 代码中提取类型信息，转化为 OpenAPI 契约；检测 Drizzle schema diff 并更新 contract 定义。此过程为结构化转换，Agent 仅做格式映射。

切入点 2: 存活变异体分析

Mutation Testing 完成后，Agent 解释存活变异体的含义，分析测试盲区原因，并建议需要补充的测试用例。人类审核最终决策。

⚖ 方案评估

TestGuard 方案的优势、劣势和适用场景评估。

优势

  • 测试质量最高 -- 变异测试是测试有效性的终极度量
  • Agent 成本极低 -- 仅占 10%，无需大量 LLM 调用
  • 100% 确定性 -- 核心流程无随机性，结果完全可复现
  • 多语言支持 -- Pact 和变异工具覆盖主流语言

劣势

  • 需要完善的类型定义 -- 契约提取依赖良好的类型系统
  • Mutation Testing 运行时间长 -- 大项目可能需数小时
  • 无法从需求直接生成测试 -- 需先有契约再有测试
  • 初始配置成本较高 -- 需设置多个工具链

适用场景

  • 对测试质量有极高要求的项目
  • 代码类型化程度高（TypeScript / Java / 有完善 Schema）
  • 微服务架构，需要强契约保证
  • 希望最小化 AI 依赖，追求确定性保障的团队
